API v2 Reference

This is the latest version of our API, adding some new functionality and some changes from v1.

In a first step, the API v2 introduces asynchronous tasks and provides new methods where the new task concept is already applied.

We will add all methods currently available in v1 to v2 at some point, likely modifiying certain details.

In the long run, support for API v1 will be terminated. We will give notice to all account owners with plenty of time in advance once we decide on a deadline for v1.

Follow the API changelog for information on added functionality.

The Basics

Sections

API methods are structured by topic. Here is the overview, including shortcuts:

  • Tasks
  • Services
  • Components

Endpoint

The default API endpoint is

https://api.giantswarm.io

For certain customers, this may vary.

Method Overview

A brief overview of all API v2 methods:

  • Task Methods
    • GET /v2/org/{org}/env/{env}/task/
    • GET /v2/org/{org}/env/{env}/task/{task}
    • GET /v2/org/{org}/env/{env}/task/{task}/waitfor
  • Service Methods
    • POST /v2/org/{org}/env/{env}/service/{service}/start
    • POST /v2/org/{org}/env/{env}/service/{service}/stop
    • DELETE /v2/org/{org}/env/{env}/service/{service}
  • Component Methods
    • POST /v2/org/{org}/env/{env}/service/{service}/component/start
    • POST /v2/org/{org}/env/{env}/service/{service}/component/stop
    • POST /v2/org/{org}/env/{env}/service/{service}/component/scaleup
    • POST /v2/org/{org}/env/{env}/service/{service}/component/scaledown

Task Methods

The task concept allows to manage asynchronous actions on Giant Swarm that can take some time to finish. Many method calls in the API v2, like for example starting a service, will return information on a task. This information can be used to track the progress of this task and wait for it to be finished.

Tasks are identified by a unique ID. Tasks can have parent and child tasks, which is indicated through references to the according IDs. It is also possible to fetch a task including all child tasks recursively.

The following task-related methods are available:

  • List tasks
  • Get Information on a Task
  • Wait For Task to Finish

List Tasks

GET /v2/org/{org}/env/{env}/task/

This method allows to list all tasks within a specific environment.

URL components:

  • {org}: Name of the organization
  • {env}: Name of the environment

Example output:

{
  "status_code": 10000,
  "status_text": "success",
  "data": [
    {
      "id": "gz0txjays0flmbc1",
      "description": "Start service bant/prod/myservice",
      "action": "start",
      "status": "finished",
      "meta": {
        "org": "tester",
        "env": "dev",
        "service": "busywait"
      }
    },
    {
      "id": "bqvyckci4wsvhlqe",
      "description": "Delete bant/prod/myservice",
      "action": "delete",
      "status": "finished",
      "meta": {
        "org": "tester",
        "env": "dev",
        "service": "busywait"
      }
    }
  ]
}

Get Information on a Task

GET /v2/org/{org}/env/{env}/task/{task}

This method allows to fetch information on a single tasks and optionally all child-tasks recursively.

URL components:

  • {org}: Name of the organization
  • {env}: Name of the environment
  • {task}: Task ID

This method supports the recursive=1 URL parameter to activate recursive retrieval of all subordinate tasks.

Example output:

      "eg5fyk5rtc8a422i"

Wait For Task to Finish

GET /v2/org/{org}/env/{env}/task/{task}/waitfor

With this method, you can fetch the result of a task. A response will only be sent once the task is finished. If the task takes a longer time to finish, this can result in the connection being closed in the meantime. In that case, the same request should simply be retried until a response with the expected JSON output is delivered (status_code equals 10000, data key contains a task object).

URL components:

  • {org}: Name of the organization
  • {env}: Name of the environment
  • {task}: Task ID

Example output:

{
  "status_code": 10000,
  "status_text": "success",
  "data": {
    "id": "0gfpijs1ghln3zdj",
    "description": "Start component redis",
    "next_id": "1owcueok4z0th1xu",
    "child_task_ids": [
      "1owcueok4z0th1xu"
    ],
    "action": "start",
    "created_at": "2015-10-09T08:17:28.651621247Z",
    "finished_at": "2015-10-09T08:17:40.604687774Z",
    "metadata": {
      "org": "bant",
      "env": "prod",
      "service": "myservice"
    },
    "status": "finished"
  }
}

Start a Service

POST /v2/org/{org}/env/{env}/service/{service}/start

This method issues a task to start a service asynchronously. The task details are returned as a result.

URL components:

  • {org}: Name of the organization
  • {env}: Name of the environment

Component Methods

The following component-related methods are available:

  • Start a Component
  • Stop a Component
  • Scale Up a Component
  • Scale Down a Component

Start a Component

POST /v2/org/{org}/env/{env}/service/{service}/component/start

This method issues a task to start a component asynchronously. The task details are returned as a result.

URL components:

  • {org}: Name of the organization
  • {env}: Name of the environment
  • {service}: Service name

The request must be sent with the Content-Type: application/json header. The request body must contain a JSON object in the following format, to indicate which component to start (replace <component_name> with the actual name of the component):

{"component": "<component_name>"}

Example output:

{
  "status_code": 10000,
  "status_text": "success",
  "data": {
    "id": "0gfpijs1ghln3zdj",
    "description": "Start component redis",
    "next_id": "1owcueok4z0th1xu",
    "child_task_ids": [
      "1owcueok4z0th1xu"
    ],
    "action": "start",
    "created_at": "2015-10-09T08:17:28.651621247Z",
    "finished_at": "0001-01-01T00:00:00Z",
    "metadata": {
      "org": "bant",
      "env": "prod",
      "service": "myservice"
    },
    "status": "pending"
  }
}

Stop a Component

POST /v2/org/{org}/env/{env}/service/{service}/component/stop

This method issues a task to stop a component asynchronously. The task details are returned as a result.

This method works exactly like starting a component, as described above in Start a component.

Scale Up a Component

POST /v2/org/{org}/env/{env}/service/{service}/component/scaleup

This method issues a task to scale up a component asynchronously. The task details are returned as a result.

URL components:

  • {org}: Name of the organization
  • {env}: Name of the environment
  • {service}: Service name

The request must be sent with the Content-Type: application/json header. The request body must contain a JSON object in the following format, to indicate which component to start:

{
  "component": "<component_name>",
  "scale": <numberof_instances_to_add>
}

The <component_name> parameter must be the full name of the component, <numberof_instances_to_add> specifies how many instances should be added for the component.

Example output:

{
  "status_code": 10000,
  "status_text": "success",
  "data":{
    "id": "4jz3dlt6xuyovx0l",
    "description": "Scale up redis",
    "next_id": "crdb5yd6rd452r5b",
    "child_task_ids": [
      "crdb5yd6rd452r5b",
      "zliily2xrhdn4ot4"
    ],
    "action": "nop",
    "created_at": "2015-10-09T09:39:38.042266868Z",
    "finished_at": "0001-01-01T00:00:00Z",
    "metadata": {
      "org": "bant",
      "env": "prod",
      "service": "myservice",
      "component": "redis"
    },
    "status": "pending"
  }
}

Scale Down a Component

POST /v2/org/{org}/env/{env}/service/{service}/component/scaledown

This method issues a task to scale down a component asynchronously. The task details are returned as a result.

This method works pretty much like the one for scaling up a component described above. Here, the scale parameter in the JSON payload specifies how many instances are to be removed.

{
  "component": "<component_name>",
  "scale": <numberof_instances_to_remove>
}

Additional Information

Date value 0001-01-01T00:00:00Z

When encountering a date/time value of 0001-01-01T00:00:00Z, regard it as an unset date field. For example, "finished_at": "0001-01-01T00:00:00Z" in task details means that there is no finished_at date set, yet.
