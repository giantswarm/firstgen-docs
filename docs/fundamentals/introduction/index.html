Introduction to Giant Swarm

With Giant Swarm you can build, deploy and manage your containerized server applications. There are no restrictions regarding programming languages, frameworks, or databases. We are especially optimized for applications that are built in a microservices style. We offer an opinionated solution, but maintain the flexibility to change the tooling to your own liking and needs.

Services you run on Giant Swarm are defined in a service definition file. In this definition you assemble any number of Docker images, each forming a component, according to your needs. To deploy a service, first, you use a Docker client or a CI/CD pipeline to create and deploy Docker images. Then you use Giant Swarm’s client software or our API to start the creation and orchestration of your Docker containers on the cluster, based on your service defintition and images.

What is Docker?

Docker is the most popular application container format. It lets you package your own stack in lightweight, portable, and self-sufficient manner. Some call it therefor lightweight virtual machine. In contrast to traditional virtual machines, Docker is usually an order of magnitudes faster to work with, due to the lack of overhead.

"Docker means self-sufficient linux containers"

Docker is a great tool for microservice developers, since it allows you to package different parts of your service in separate images. These images can be executed as containers on different environments like a personal laptop as well as different server environments, like Giant Swarm’s clusters.

Docker benefits from a large, ever growing ecosystem with lots of predefined images for all sorts of tasks. From different languages like JavaScript to Golang over different stacks like Spring to Rails down to databases like Redis or MySQL. This makes defining individual images, based on templates from the ecosystem, very easy.

The Anatomy of a Giant Swarm Service

A service on Giant Swarm is a named grouping of any number of service components (or briefly components), where each component usually represents a Docker image. As a Giant Swarm user, you can define any number of services and it is up to you to decide how big or small your service(s) should be.

Service components can be linked to each other, so that connections can be made easily, for example from an application server component to a cache component. These links work analog to Docker links. Each component can have multiple Docker containers running it, which we call component instances (or just instances for short). Giant Swarm’s orchestration automation takes care for you that load is distributed between instances. You can use Giant Swarm’s software or API to monitor the activity and resource usage of your instances and scale components to cope with fluctuating loads.

A rather simple real-life example for a service on Giant Swarm would be a web application that consists of a caching proxy component, a webserver component, a web application component, caches, data stores, and a search engine.

Or, if you’d like to follow a domain-driven approach, you could define e. g. a product catalog service next to a shopping cart service. Both could consists of similar components like web application servers, caches, and databases. Both services could have connections to each other.

The Service Definition

For Giant Swarm, services are defined in a JSON-based definition format. The typical service definition file is called swarm.json and consists of two main elements: A name and a set of components. The components are the building blocks of services on Giant Swarm.

Let’s look at an example. The following service named shop consists of three components: A main application server (called main), a Redis data store (redis) and an Elasticsearch server (elasticsearch). The main component is publicly accessible via a domain, and it is connected internally to the other components via links.

{
  "name": "shop",
  "components": {
    "main": {
      "image": "registry.giantswarm.io/acme/shop-main",
      "ports": 8000,
      "domains": {
        "8000": "acme-shop.com"
      },
      "links": [
        {"component": "redis", "target_port": 6379},
        {"component": "elasticsearch", "target_port": 9200}
      ]
    },
    "redis": {
      "image": "redis:2.6",
      "ports": 6379
    },
    "elasticsearch": {
      "image": "elasticsearch:1.7",
      "ports": 9200
    }
  }
}

Controlling Giant Swarm Services

Your main interface to Giant Swarm is a command line interface (CLI) called swarm. It allows you to deploy services based on a service definition, start, stop and scale components, fetch status information, statistics, and logs.

$ swarm ls
2 services available:

service         environment  created              status
currentweather  luebken/dev  2014-11-28 14:01:35  up
helloworld      luebken/dev  2014-11-28 14:01:43  up

In addition, we are working on a complementing web user interface, allowing for simple and fast status checks, log access, and statistics monitoring.

For deeper integration into custom-built tool chains, the Giant Swarm API, which is used by the CLI and web UI software, is open to all users.

To facilitate team collaboration, Giant Swarm makes all services belong to an organization. Users can add and remove users to organizations to form teams with access to all services within the respective organization.
