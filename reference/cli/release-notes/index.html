Release Notes

Here we list the more important changes we made between releases. See our installation reference for information on how to update to the latest CLI.

Version 0.27.0

Released 2016-04-01

  • Added support for shared volumes. See the volumes section in the Service Definition reference to learn more.

Version 0.26.0

Released 2015-12-21

  • Added activity tracking, which means that the command line content and an activity name will be send with every API call. Set the environment variable SWARM_DISABLE_CMDLINE_TRACKING to disable sending the command line content. (cli#534)
  • Fixed a problem in swarm create where the context variable $GIANTSWARM_ENVIRONMENT was parsed as $GIANTSWARM_ENV (cli#498).
  • Fixed a problem where swarm update <service>/<component> didn’t work when a file ./<service> existed (cli#527)
  • Improved feedback for swarm org when trying to add a user to an organisation that either doesn’t exist or that the user has no access to (cli#349)
  • When calling swarm ls -a, all requests made will have the same request ID (#503)
  • More tolerant behavior of swarm update for components, allowing to set the second argument as <image>:<tag> (cli#378)
  • Improved output for swarm update when updating a service in the case that the validation yields an error (cli#481)

Version 0.25.0

Released 2015-11-30

  • If swarm logs returns an empty result, a short message is displayed. This can be suppressed using -q (cli#4)
  • Activating the --debug flag will now attempt to print the URL of all API requests made. (cli#385)
  • Better feedback in case of an invalid API endpoint setting (cli#441)
  • Better feedback when fetching status for a non-existing component (cli#499)
  • Better feedback for swarm start and swarm stop when the component does not exist (cli#502)
  • Better feedback for swarm up when service already exists (cli#372)
  • Better feedback for swarm exec when instance used is down (cli#467)
  • Better feedback when service definition contains link using undefined port (cli#512)
  • When not in an interactive terminal context, the progress indicator for commands is simpler (cli#420)
  • When not in an interactive terminal, the update notification is suppressed (cli#412)
  • The CLI is now built using Go verison 1.5.1.

Version 0.24.1

Released 2015-11-24

  • Fixed invalid link cycle errors when component A links to component B and component B links to another service (user-config#80)

Version 0.24.0

Released 2015-11-11

  • swarm update for a component should no longer exit with an EOF error. (cli#366)
  • Gateway timeout errors breaking long-running tasks should now be avoided (cli#472)
  • When asynchronous tasks finish with an error, that error is now returned to the CLI (cli#454)
  • swarm info now shows the cluster ID if the client has one selected (cli#436)
  • More friendly handling of errors and special cases, e. g.
    • Attempt to start a service in a non-existing environment (cli#362)
    • Attempt to execute swarm logs with an argument that is not a valid instance ID (cli#371)
    • Attempting to create an organization that already exists (cli#351)
    • Attempting swarm update for an unknown/misspelled component name (cli#399)
    • Attempting swarm scaleup or swarm scaledown for an unknown/misspelled component name (cli#455)
  • Enabled the image name truncation in swarm status output again (cli#377)

Version 0.23.0

Released 2015-10-30

  • The service definition now supports a memory-limit directive per component. This is only applicable on certain clusters, not on the Shared alpha cluster. More details
  • swarm info now also prints the API endpoint. Also the output order has changed. (cli#442)
  • swarm stop no longer shows text related to starting a service in case of an unknown service name (cli#443)

Version 0.22.1

Released 2015-10-13

  • Enhanced detection of connection problems, which should result in better robustness when waiting for the result of a long-running task (client-go#81).
  • Fixed a stupid bug were swarm stop on a component would result in the attempt to start the component instead of stopping it (cli#433).

Version 0.22.0

Released 2015-10-09

  • The commands start, stop, scaleup, scaledown, and delete use new asynchronous API methods, which should prevent connection timeouts (known as EOF errors or unknown server errors) from happening.
  • Changes to swarm update:
    • When updating a service definition, you will now get a detailed overview of the changes to be applied, including warnings where we think you should be aware of the consequences.
    • Updating a service definition no longer allows to add the -s/--strategy flag to select an update strategy, since the update strategy when updating services is always hot-swap.
  • Changes to swarm delete:
    • While waiting for the deletion to be finished, we now show an activity indicator indicator.
    • The command now provides the -d flag to return immediately after issuing the command, not waiting for the result.
  • swarm scaleup now waits for the result of the scaling action and shows an activity indicator in the meantime.
  • swarm scaledown now also shows an activity indicator while waiting for the result.

Version 0.21.0

Released 2015-09-17

  • Fixed panics in the CLI when using swarm update on a service using the expose functionality to link different components together (components without an image field) user-config#65 #406
  • Removed swarm migrate command

Version 0.20.0

Released 2015-09-14

  • The new placement key in scaling policy allows for one-per-machine placement of component instances, e.g. for high availablity setups.
  • The new component level key signal-ready allows for manually signaling readiness from components. By setting signal-ready to true you set a component to manual readiness notification. In this case you have to use the environment variable GIANTSWARM_INSTANCE_API, which contains a URL endpoint as a value, to set the status to ready with a simple HTTP POST request in your component.
  • Added ability to update services based on changes in the service definition with swarm update. Note: This feature is an early iteration and thus not documented, yet. We will release proper documentation of this feature, once it’s ready for public usage.

Version 0.19.1

Released 2015-08-14

  • Added functionality that simplifies the debugging of requests. If you set the environment variable SWARM_REQUEST_ID to some string value, this string is used as the prefix for an automatically generated request ID, which is then sent to the API as an HTTP header.

Version 0.19.0

Released 2015-08-10

  • What used to be called application is now called service. The way services are defined has been changed heavily. Read more in the Service Definition (swarm.json) Reference. Some background about the motivation behind this move is given in the blog post Introducing A Better Way To Define Services.
  • The commands swarm status, swarm start and swarm stop now support a new flag -r or --recursive, that allows to recursively deal with all components starting at a certain point in the component hierarchy.
  • swarm start and swarm stop allow to start/stop individual components now.
  • The swarm up and swarm create commands now allow to set the name of the new service via the command line. This overwrites a name in the service definition, if given.
  • The new command swarm migrate has been introduced to migrate old application configuration files (swarm.json) to the new service definition format.
  • The service definition now supports the use of predefined variables. For example, $GIANTSWARM_LOGIN_NAME will be translated to the current username. More details
  • For users running docker 1.7 or later, swarm login will now automatically log users in to our registry (registry.giantswarm.io). swarm logout will correspondingly log users out of our registry too. Authentication tokens are written to ~/.docker/config.json
  • The username of the currently authenticated user is now stored to a local file. The default path for this file is ~/.swarm/username, which can be modified using the global --config-dir flag.
  • The positioning of flags no longer matters, so that swarm -v status will work just as good as swarm status -v.
  • The output of swarm status now truncates the image column to make the output more compact. The name of our registry registry.giantswarm.io is truncated to reg…. Use the -v or --verbose flag to show untruncated output.

Version 0.18.0

Released 2015-06-22

  • There’s a new component-level configuration key pod, with which you can group several components for sharing namespaces between them. You can read more about it on the swarm.json reference page.
  • Above mentioned pod key allows for the new volumes-fromand volume-from keys under the volumes key. Read more on the respective swarm.json reference page.
  • You can now choose update strategies for swarm update with -s <update-strategy>. Choose between one-by-one, all-at-once, and hot-swap. Read more on the update reference page.
  • swarm is now compiled statically, making it possible to run inside boot2docker (e.g. on a Windows machine).
  • The new --config-dir option has been introduced, to set custom configuration directories. Read more on the global options reference page.

Version 0.17.0

Released 2015-05-11

  • The new function swarm exec allows to execute arbitrary commands in a running component instance, including opening a shell. Read more about it on the according reference page.
  • For any component that creates log output, log data is now reliably retained for a fixed period (currently 14 days) as long as the source component instance still exists. Deleting a component instance makes the according log data inaccessible.
  • This also results in a change in the swarm logs mechanics: the flags -t and -f can no longer be combined. The new implementation of historic log data access makes this difficult to achieve. Once real-time log access is unified, this function might return.
  • The new component-level configuration key entrypoint has been introduced, allowing to override a Docker image’s ENTRYPOINT directive. Read more in the according Application configuration section.
  • Fixed a bug that broke the function of swarm logs <instance-id> -f.
  • Fixed a bug that slowed down the execution of calls to swarm status and swarm stats.

Version 0.16.0

Released 2015-04-20

  • What was called “company” so far is now called an “organization”. The swarm company CLI command has been renamed to swarm org.
  • The swarm update command is back and allows to update the docker image run by a component. Check the reference page for details.
  • The swarm stats command now allows showing the stats for more than one instance. it can be called with an application, service or component name as a selector. Check the reference page for details.
  • The swarm ls command can now list applications from all organizations and environments accessible for the current user, using the --all/-a flag (swarm ls -a). Also the command executes faster than before due to parallelization.
  • Commands which accept an instance ID as argument (swarm logs, swarm stats) now also accept partial instance IDs, as long as this part is not ambigious. If, for example, an instance ID is 3nFK77aEF88NnSww, it will in most cases suffice to call its stats using swarm stats 3n or even swarm stats 3.
  • The swarm logs command now defaults to showing the 10 latest rows. Use -t <n> to show a specific number of entries or -t all to show all entries. See the reference page for details.
  • We now publish SHA1 and MD5 checksums/hashes for our CLI release files. Check the install reference for details.

Version 0.15.0

Released 2015-02-24

  • We introduced version checks between CLI and API. Deprecated clients now show an error and prompt the user to install an update whenever the CLI is no longer compatible.
  • The CLI now checks once per day for a CLI update and informs the users if a newer version is available.
  • Password changes now result in the user being logged out.
  • Fixed a problem with continuous log output (-f switch in swarm logs).

Version 0.12.0

Released 2015-02-09

  • swarm status now prints the Docker image used by each component.
  • When an error occurs during stopping a component (swarm stop or swarm delete), this error is displayed to the end user. A common cause for errors when stopping is the improper handling of the stop signal (SIGTERM). When you see this kind of error, make sure your containers handle this signal properly.
  • The parsing of a local application configuration file (swarm.json) has improved so that the use of commands like swarm status, swarm start and swarm stop is possible without using an application name as argument, even when there are variables in the configuration file.
  • swarm info and swarm version now check whether the current CLI version is the latest. In addition, the CLI now sends the CLI version with every request to the API. With this, we are working towards automatic compatibility checks and warnings.
  • Port definitions in the application configuration now accept integer values
  • Several fixes and improvements of robustness.

Version 0.10.2

Released 2015-01-13

  • We added a simpler, more straight-forward way to define environment variables in the application configuration (swarm.json). They can now be set as an object like this: "env": {"KEY_ONE": "value_one", ...}. The old Array format will still work, but all documentation has been updated to use the new format.

Version 0.10.0

Released 2015-01-09

  • The swarm update has been deactivated due to possible problems and side-effects until we come up with a more stable implementation. Updating components requires the use of swarm stop and swarm start.

  • The swarm company command has changed in several ways. swarm company -l now lists all companies for the user. swarm company create <companyname> creates a new company.

  • The swarm scaleup and swarm scaledown commands have changed. They can now be used with only a component name as argument and the number 1 will be assumed as default number to scale by. The named argument --count can no longer be used.

  • swarm status no prints the instance creation date for every instance. (Instances created before the release show n/a here.)

  • The output for swarm ls has changed slightly for consistency reasons. The environment is now printed in one column in the common format <company>/<env>.

  • Fixed a bug in the client where certain passwords where wrongly passed to the API for authentication.

  • swarm help <command> now prints documentation URLs in some cases. CLI usage texts have been shortened in some cases.

  • swarm user without argument now prints the currently logged in user.

  • Messages printed by some commands in case of errors has been improved.

  • New subcommands have been incorporated into the bash completion configuration.

  • Changing the email address via swarm user -u email no longer requires to enter the old email address.

  • swarm info no longer shows a “Current company”, since there actually is no such thing. There was and is a “Current environment” however, which automatically belongs to a company, as indicated by its name.

  • swarm env <environment-name> now checks access to the environment (company membership) before setting it as the current one.

  • Fixed a bug where the progress indicator in commands like swarm up was hidden behind the cursor in some terminals and another bug causing the progress indicator to look “unround”.
